Part I:
Cryptographic Primitives

The "crypto" in cryptocurrency: hashes, keys, signatures, and addresses.

03

why_crypto.md

> No central accountant. So the data has to prove itself.

  • hashes make data tamper-evident
  • signatures prove authorization
  • nodes verify the rules independently

A blockchain record should be:

[ easy to verify ] | [ hard to forge ] | [ hard to rewrite quietly ]

The system replaces “trust the institution” with “verify the data.”

04

hash_functions.exe

  • same input → same hash
  • tiny input change → totally different hash
  • hard to reverse
  • fixed-size output

A hash is a fingerprint, not encryption. It identifies data; it does not hide data.

[ INPUT DATA ] "hello" | V +-----------+ | HASH | | FUNCTION | +-----------+ | V [ OUTPUT HASH ] 2cf24dba5fb0...
05

find_hashes.sh

Transaction ID

A hash of the transaction data itself.

Block Hash

A hash of the entire block header.

Merkle Root

One fingerprint representing all transactions in a block.

Hashes let the network detect when transaction or block data has been changed.

06

merkle_tree.log

One hash for many transactions.

[ Merkle Root ] / \ / \ [ hAB ] [ hCD ] / \ / \ / \ / \ [ hA ] [ hB ] [ hC ] [ hD ] | | | | [TX A] [TX B] [TX C] [TX D]

A Merkle root commits to every transaction in the block without listing them all in the header.

07

generate_keys.py

[ PRIVATE KEY ] (secret) | V +-----------+ | ELLIPTIC | | CURVE | | MATH | +-----------+ | V [ PUBLIC KEY ] (shareable)
  • Easy direction: private → public.
  • Hard direction: public → private.

The private key is a massive random number. The public key is derived from it using one-way cryptography.

08

sign_tx.exe

Authorization without revealing the key.

The signature proves the private key authorized this transaction. The private key itself is never revealed on the network.

[ TX MESSAGE ] + [ PRIVATE KEY ] \ / \ / \ / +-------------+ | [SIGN] | +-------------+ | V [ SIGNATURE ]
09

verify_sig.exe

[ TX MESSAGE ] + [ SIGNATURE ] + [ PUBLIC KEY ] \ | / \ | / \ | / +-------------------------+ | [VERIFY] | +-------------------------+ | V [ VALID / INVALID ]

Anyone can check the signature.

  • Signing requires the private key.
  • Verification only needs the public key.
10

encode_address.sh

Destinations, not identities.

  • An address is a shareable way to receive value.
  • It is not a name.
  • It is not proof of identity.
[ PUBLIC KEY / SCRIPT ] | V +-------------------+ | Hash + Encoding | | + Checksum | +-------------------+ | V bc1q...7f3 (The Address)
11

derive_hd_wallet.py

[ SEED PHRASE ] (12/18/24 words) | (derive keys) | +---------+---+---+---------+ | | | | [Addr 1] [Addr 2] [Change] [Addr n]

HD = Hierarchical Deterministic.

One seed can generate many keys and addresses.

This is why wallets can generate fresh change addresses without requiring a new backup every time.

12

control_model_recap.txt

If you understand this control model, Bitcoin and Ethereum become much easier to read.

Private Key

The secret. It can sign transactions.

Signature

The mathematical proof that authorizes a specific transaction.

Public Key

The visible key that the network uses to verify the signature.

Address

A hashed version of the public key where value is locked / received.

Part II:
Bitcoin Under the Hood

Outputs, UTXOs, inputs, fees, change, and signatures.

14

view_models.cfg

User View (GUI)

  • I have a wallet balance.
  • I send 0.10 BTC to Bob.
  • The wallet shows a new balance.

Technical View (CLI)

  • Wallet selects UTXOs.
  • Transaction consumes inputs.
  • Transaction creates new outputs.
  • Signature proves authorization.

The explorer may make Bitcoin look account-based. Under the hood, it is output-based.

[ DOES NOT ]

17

execute_send.sh

What the wallet actually does when you click "Send":

[1. SELECT UTXOs] ---> [2. BUILD TX] ---> [3. SIGN INPUTS] | V [5. CONFIRM IN BLOCK] <--- [4. BROADCAST TO NETWORK]

The send button hides output selection, change creation, signing, broadcast, and confirmation.

18

network_relay.log

Broadcast and Relay

A transaction enters one node, then propagates across the peer-to-peer network via gossip protocol.

[Wallet TX] | V (Node 1) ---> (Node 2) | \ | V \ V (Node 3) +-> (Node 4) | | V V (Node 5) <--- (Node 6)
19

node_consensus.exe

A node checks:

  • Inputs refer to unspent outputs
  • Signatures are valid
  • Transaction follows consensus rules
  • No double spend

A node rejects:

  • Overspending
  • Invalid signatures
  • Spent outputs
  • Malformed transactions

Nodes do not ask who you are. They ask whether the transaction is valid.

20

block_assembly.py

[ MEMPOOL ] [ MINER ] [ BLOCKCHAIN ] (Pending TXs) ========> (Builds Block) =======> (Adds to Chain)

A confirmation means the transaction is included in a block.
More blocks on top = harder to reorganize.

21

tx_outputs.json

Locked chunks of value.

  • Bitcoin value lives in outputs.
  • An address is the lock placed on an output.
{ "Output": { "amount": "0.50 BTC", "lock_script": "prove you have the right private key" } }
22

utxo_set.db

[ TX 1 ] |-- (spent) | +-- [ UTXO A: 0.30 BTC ] [ TX 2 ] |-- [ UTXO B: 0.70 BTC ] | +-- [ UTXO C: 0.15 BTC ]

The spendable tips of the graph.

A UTXO (Unspent Transaction Output) is simply an output that has not been spent yet.

23

tx_inputs.json

Inputs point backward.

An input does not contain coins. It points to a previous output and unlocks it.

[ PRIOR TX OUTPUT ] (0.80 BTC) ^ | (references) | [ NEW TX INPUT ] | V [ NEW OUTPUTS CREATED ]
24

tx_anatomy.sys

Consume and create.

+-------------------+ +-----------------------+ +-------------------+ | Input 1 (UTXO) | | | | Output 1 (Recip) | | 0.35 BTC |----->| BITCOIN |----->| 0.40 BTC | +-------------------+ | TRANSACTION | +-------------------+ | | +-------------------+ | Total In: 0.55 | +-------------------+ | Input 2 (UTXO) |----->| |----->| Output 2 (Change) | | 0.20 BTC | +-----------------------+ | 0.149 BTC | +-------------------+ +-------------------+ [!] Fee = Inputs - Outputs (0.55 - 0.549 = 0.001 BTC)
25

calculate_fee.py

fee = inputs - outputs

Bitcoin fees are implicit. There is no separate "fee output" recorded on the ledger.

> let inputs = 0.35 + 0.20 > let outputs = 0.40 + 0.149 > print(inputs) 0.55 BTC > print(outputs) 0.549 BTC > print(inputs - outputs) 0.001 BTC (Miner Fee)
26

change_mechanics.md

[ UTXO: 1.00 BTC ] | | (consumed fully) V [ TX ] / \ / \ [PAY] [CHANGE] 0.25 0.749 (Fee: 0.001)

Why wallets create another output:

  • You cannot spend part of a UTXO.
  • The whole output is consumed.
  • New outputs are created for the destination and the change.
27

unlock_script.exe

Signatures unlock prior outputs.

The network verifies: does the unlocking data (signature) satisfy the locking condition (address)?

[ LOCKED OUTPUT ] (scriptPubKey) ^ | [ INPUT PROVIDES ] (sig + key data) | V [ VALID SPEND ]

Ethereum Under the Hood

Accounts, gas, smart contracts, and token events.

31

state_machine.sys

Ethereum is a shared state machine.

[ STATE BEFORE ] =======> ( TRANSACTION ) =======> [ STATE AFTER ]

Ethereum transactions change account balances and contract state directly.

32

account_types.cfg

Externally Owned Account (EOA)

  • Controlled by a private key
  • Can initiate transactions
  • Example: MetaMask account

Contract Account

  • Controlled by code
  • Has storage and logic
  • Executes when called

Same address format. Very different control model.

33

interface_layer.md

Wallets are interfaces; accounts are on-chain objects.

A wallet is the tool. An account is what actually exists on the blockchain state.

[ WALLET APP ] (MetaMask / Rabby) | | (manages) V [ PRIVATE KEY / SEED ] | | (controls) V [ EOA ADDRESS ] (0x... on-chain)

Code at an address.

erc20_ledger.db

Ledgers inside contracts.

38

transfer_comparison.md

Simple ETH Transfer

[Address A] --(5.0 ETH)--> [Address B] * Gas deducted from A separately. * Recipient does not pay the gas fee.

ERC-20 Token Transfer

[Address A] --(TX)--> [USDT Contract] | (Internal Ledger) A: -10,000 USDT B: +10,000 USDT | (Transfer Logged) V [Address B]

The transaction's "To" field shows the contract address, not the token recipient.
The actual token movement is logged as an event inside the contract.

39

read_logs.sh

Parent Transaction

  • from: caller EOA
  • to: token contract
  • value: often 0 ETH
  • gas paid by caller

Token Event

  • from: token sender
  • to: token recipient
  • amount: token value
  • emitted by contract

A beginner reads the parent tx. An analyst reads the event logs too.

set_allowance.exe

[ OWNER ] (Address A) | | approve(spender, amount) V [ CONTRACT ] | | (allowance updated) V [ SPENDER ] (Can now call transferFrom)

Approvals and allowances.

Approval grants permission to another address or contract. It does not move tokens yet.

drain_sequence.log

=== STEP 1: APPROVAL TRANSACTION (Victim pays gas) === [Phishing Site] -> [Victim EOA] --(approve tx)--> [USDT Contract] | [Attacker Contract] <--(Granted Spending Permission)-+ (NO TOKENS MOVE YET) ... Hours or days later ... === STEP 2: DRAIN TRANSACTION (Attacker pays gas) === [Attacker EOA] --(calls contract)--> [Attacker Contract] | (transferFrom) V [Victim EOA] <--(Ledger Update)---- [USDT Contract] --(Ledger Update)--> [Attacker Address] (-10,000 USDT) (+10,000 USDT)

The victim only signed once (the approval). The drain happens later without any action from the victim.

Part IV:
Compare the Models

Same cryptography; different ledger mechanics.

45

diff models.txt

Bitcoin (Outputs)

  • Outputs are value chunks
  • Inputs spend prior outputs
  • Change is created
  • Fee = inputs - outputs

Tracing asks: Which output continues?

Ethereum (State)

  • Accounts have balances
  • Transactions change state
  • Gas pays for execution
  • Token movement is events/state

Tracing asks: What state/event changed?

46

error_logs.txt

  • Treating a Bitcoin address like a bank account
  • Assuming the larger Bitcoin output is always change
  • Missing ERC-20 token transfers because ETH value is zero
  • Reading the transaction “to” field as the token recipient
  • Forgetting gas/native asset is separate from token value

The fix is the same every time: slow down and inspect the actual mechanics.
